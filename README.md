## 🎯 Цель проекта

Мини-прототип мобильной FPS-игры с фокусом на **быстрый и понятный gameplay loop**.

Игрок взаимодействует с объектами в мире, запускает процессы на станциях (например приготовление еды) и должен **следить за несколькими задачами одновременно**, реагируя на изменения их состояния.

Основная цель разработки — за короткое время (1–2 дня) собрать **рабочий, понятный и расширяемый прототип**, уделяя внимание ощущению взаимодействия и читаемости происходящего.

## 🧩 Реализовано

### Основной геймплей
- Подбор предметов
- Перенос предмета в руках
- Использование предмета на объектах
- Система взаимодействия через raycast от камеры

### Станции

Система станций полностью конфигурируется через **ScriptableObject**.

Для каждой станции задаётся:
- входной предмет
- тайминги (Cook / Warning / Fail)
- результат при успехе
- результат при ошибке

Это позволяет добавлять новые типы станций и задач **без изменения кода**, только через конфигурацию.

---

### Поведение при ошибке

Даже при достижении стадии Fail станция всё равно выдаёт результат  
(например: "сгорело" или "холодный кофе").

Это сделано осознанно:

- игрок не блокируется
- сохраняется ритм геймплея
- появляется вариативность (разные исходы)
- уменьшается фрустрация

Такой подход ближе к реальным cooking / management играм, где ошибка — это не стоп, а альтернативный результат.

---

### UI и визуальный фидбек
- World-space индикаторы прогресса
- Цветовая индикация стадий:
  - белый — готовится
  - жёлтый — warning
  - красный — ошибка
  - зелёный — готово
- Панель активных задач
- Hover-подсказки взаимодействия

---

### Управление (Mobile)
- виртуальный джойстик для движения
- свайп для поворота камеры
- поддержка multitouch
- разделение зон экрана (движение / камера / UI)

---

### Фидбек
- звуковые события станций
- частицы для приготовления и ошибки
- визуальные сигналы стадий

## 🏗 Архитектура

Проект разделён на логические модули:

- **Core**
- **Gameplay**
- **Systems**
- **UI**

Используется **data-driven подход через ScriptableObject**.

Добавление новых:
- предметов
- станций
- рецептов

происходит через конфигурацию, **без изменения кода**.

Архитектура намеренно упрощена, чтобы сохранить **скорость разработки и читаемость систем**, без избыточного усложнения.

## 🤖 Использование AI

AI использовался как вспомогательный инструмент для ускорения разработки и проверки решений.

Основные задачи, где он применялся:

- обсуждение архитектуры систем
- генерация базовых каркасов классов
- помощь в поиске и устранении ошибок
- быстрые итерации при улучшении UX
- оптимизация мобильного ввода и multitouch логики

Примеры задач:

- проектирование системы предметов и взаимодействия
- упрощение логики станций без сложной state machine
- реализация data-driven конфигураций через ScriptableObject
- улучшение обработки мобильного ввода
- идеи для визуального и звукового фидбека
  
Примеры промптов:
- "Спроектировать WorldItem + Interaction + Player Carry flow"

- "Сделать ScriptableObject конфиг станции с входными предметами и таймингами"

- "Реализовать StationBehaviour без сложной state machine, через enum"

- "Добавить hover-методы в IInteractable для подсказок"

- "Переписать Update так, чтобы переходы состояний не вызывались каждый кадр"

AI позволил сократить время на рутинные задачи и сосредоточиться на **геймплейной логике и ощущении взаимодействия**.

## ⚠️ Какие слабые места есть в текущем решении

Поскольку это быстрый прототип (1–2 дня разработки), часть решений была сознательно упрощена.

Основные слабые места текущей реализации:

### Упрощённая логика станций
Система состояний реализована через enum и таймеры.  
Для прототипа это достаточно, но при росте сложности логики (больше стадий, зависимостей или параллельных процессов) могла бы понадобиться более формализованная state machine.

### Мобильный ввод
Система управления реализована вручную через обработку touch-событий.  
Она решает основные задачи (multitouch, разделение зон экрана), но при дальнейшем развитии стоило бы:
- добавить более плавное сглаживание камеры
- улучшить обработку edge-case ситуаций с несколькими пальцами
- нормализовать чувствительность для разных устройств.

### Минимальный визуальный полиш
Основной упор был сделан на читаемость геймплея и фидбек, но визуальная часть может быть значительно улучшена:
- больше анимаций
- richer VFX
- улучшенные UI элементы
- более выразительная визуальная идентичность станций.

### Отсутствие полноценного gameplay progression
Сейчас прототип демонстрирует базовый gameplay loop и систему станций, но не содержит:
- progression
- score системы
- усложнения задач со временем.

---

Эти ограничения являются следствием фокуса на **скорости разработки и демонстрации базовых игровых систем**, а не на полном продакшн-качестве.
